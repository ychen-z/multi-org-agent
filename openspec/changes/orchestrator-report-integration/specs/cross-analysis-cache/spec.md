## 新增需求

### 需求: MongoDB 缓存集合

系统必须在专用的 MongoDB 集合中存储交叉分析结果，并支持自动过期。

#### 场景: 缓存集合结构

- **当** 创建 analysis_cache 集合
- **则** 文档结构为：{\_id, type, data, created_at, expires_at}
- **且** 在 expires_at 字段上创建 TTL 索引

#### 场景: TTL 索引自动清理

- **当** 文档的 expires_at 时间到期
- **则** MongoDB 自动删除该文档
- **且** 无需手动清理

### 需求: 缓存键生成

系统必须基于分析类型和时间窗口生成确定性的缓存键。

#### 场景: 交叉分析缓存键

- **当** 执行 cross_analysis()
- **则** 缓存键生成格式为 "cross*analysis*{YYYY-MM-DD-HH}"
- **且** 同一小时内生成相同的键（1小时粒度）

#### 场景: 按部门筛选的缓存键

- **当** 执行 cross_analysis(department_id="DEPT001")
- **则** 缓存键包含部门："cross*analysis*{日期}\_{department_id}"

### 需求: 计算前先查询缓存

系统必须在执行耗时的交叉分析查询前检查缓存。

#### 场景: 缓存命中

- **当** 调用 cross_analysis()
- **且** 当前时间窗口存在有效的缓存条目
- **则** 系统直接返回缓存数据，不执行 MongoDB 查询
- **且** 响应时间 < 100ms

#### 场景: 缓存未命中

- **当** 调用 cross_analysis()
- **且** 不存在有效的缓存条目
- **则** 系统执行完整的交叉分析查询
- **且** 将结果存入缓存，TTL 为 1 小时

### 需求: 缓存失效

系统必须支持手动缓存失效。

#### 场景: 强制刷新参数

- **当** 调用 cross_analysis(force_refresh=True)
- **则** 系统跳过缓存查询
- **且** 执行新查询
- **且** 用新结果更新缓存

#### 场景: 数据导入时清除缓存

- **当** 通过 /api/v1/data/import 成功导入新数据
- **则** 系统清除所有 analysis_cache 条目
- **且** 下次分析使用新数据

### 需求: 缓存 TTL 配置

系统必须支持可配置的分析缓存 TTL。

#### 场景: 默认 TTL

- **当** 未指定 TTL
- **则** 缓存条目在 1 小时（3600 秒）后过期

#### 场景: 通过配置自定义 TTL

- **当** 设置 ANALYSIS_CACHE_TTL 环境变量
- **则** 缓存条目使用配置的 TTL 值

### 需求: 缓存统计

系统必须跟踪缓存命中/未命中统计。

#### 场景: 记录缓存操作日志

- **当** 访问缓存
- **则** 系统记录是命中还是未命中
- **且** 记录使用的缓存键

#### 场景: 通过 API 暴露缓存统计

- **当** 客户端请求 GET /api/v1/system/cache-stats
- **则** 系统返回 {hits, misses, hit_rate, entries_count}
