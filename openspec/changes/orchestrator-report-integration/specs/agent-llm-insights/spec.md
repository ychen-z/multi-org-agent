## 新增需求

### 需求: 交叉分析洞察生成

系统必须在交叉分析数据收集完成后调用 LLM 生成智能洞察。

#### 场景: 生成交叉分析洞察

- **当** cross_analysis() 完成，包含 recruitment_performance、performance_turnover、manager_team_impact 数据
- **则** 系统调用 LLM，传递包含数据的结构化 prompt
- **且** LLM 返回 JSON 格式的洞察，包含关键发现和建议

#### 场景: 洞察 prompt 结构

- **当** 调用 LLM 生成交叉分析洞察
- **则** prompt 包含系统角色为"HR 数据分析专家"
- **且** prompt 包含结构化的数据摘要
- **且** prompt 要求特定的洞察类别（模式、异常、建议）

### 需求: 通过 LLM 生成 CEO 报告

系统必须使用 LLM 和所有收集的分析数据生成最终的 CEO 战略报告。

#### 场景: 生成 CEO 报告

- **当** 所有子 Agent 分析和交叉分析完成
- **则** 系统调用 LLM 生成完整的 Markdown 报告
- **且** 报告包含执行摘要、关键指标、洞察和行动项

#### 场景: 报告 prompt 结构

- **当** 调用 LLM 生成 CEO 报告
- **则** prompt 包含所有分析结果作为上下文
- **且** prompt 指定 CEO 级别的语言风格和可执行建议
- **且** 输出格式为 Markdown，包含清晰的章节

#### 场景: 报告包含必需章节

- **当** 生成 CEO 报告
- **则** 报告包含：执行摘要、关键指标、风险分析、交叉洞察、建议（短期/中期/长期）、行动清单

### 需求: LLM 失败时的回退处理

系统必须在 LLM 失败时优雅地使用规则回退方案。

#### 场景: LLM 超时

- **当** LLM 调用超过 30 秒
- **则** 系统取消调用
- **且** 使用规则模板生成报告（不包含 AI 洞察）
- **且** 报告包含提示："AI 洞察生成失败，使用规则报告"

#### 场景: LLM API 错误

- **当** LLM 返回错误（限流、认证失败等）
- **则** 系统记录错误日志
- **且** 使用规则模板生成报告
- **且** 错误不影响报告交付

### 需求: 可配置的 LLM 提供商

系统必须使用配置的 LLM 提供商生成报告。

#### 场景: 使用默认提供商

- **当** 未为报告生成配置特定提供商
- **则** 系统使用设置中的默认 LLM 提供商

#### 场景: 使用指定提供商

- **当** 请求中包含 provider 参数
- **则** 系统使用指定的提供商（openai/qwen/glm/ollama）

### 需求: 优化 LLM 调用次数

系统必须最小化 LLM 调用次数以控制成本和延迟。

#### 场景: 限制 LLM 调用

- **当** 执行完整报告生成
- **则** 系统最多进行 3 次 LLM 调用：
  1. 交叉分析洞察
  2. CEO 报告生成
  3. （可选）行动清单优先级排序
